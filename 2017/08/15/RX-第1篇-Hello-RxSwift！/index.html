第1篇 Hello RxSwift！

说了这么多，到底什么是RxSwift？
有以下定义：

RxSwift is a library for composing asynchronous and event-based code by using observable sequences and functional style operators, allowing for parameterized execution via schedulers.

我来翻译下：

RxSwift是通过使用可观察序列和功能样式运算符来组合异步和基于事件的代码的库，允许通过调度程序进行参数化执行。

？？？这是啥啥
（这句话浓缩了很多东西，其实是写给大神看的

写这篇文章的目的，就是要向你一点点介绍RxSwift的API，还有响应式编程的概念。

来一个简化版的定义：

RxSwift, in its essence, simplifies developing asynchronous programs by allowing your code to react to new data and process it in sequential, isolated manner.
RxSwift本质上通过让你的代码可以对新数据做出反应并以顺序、隔离的方式处理数据来简化开发异步程序。

这个定义对于一个初学者来说就更简单易懂了。

异步编程

这个东西其实我们在以前就经常用到了。对于一个简单的iOS App，在同一时间，可能会对以下事件进行响应：

  • 响应对Button的点击操作
  • 显示当text field失去焦点时的键盘消失动画
  • 从网络上下载一张图片
  • 向磁盘中写入一些数据
  • 播放提示音

这可能只是其中的一部分，但这些事件确实会在同一时间发生。但是这些不同的事件之间并不会互相阻碍（当然这利用了多线程技术）。

写异步代码确实不是一件简单的事，而且当这些代码要对同一块数据进行操作时，就更复杂了。

关于Cocoa和UIKit的异步API

为了解决以上的问题，Apple已经向开发者提供了很多的API，比如：

  • NotificationCenter：通过通知机制来响应比如键盘弹出收回、屏幕旋转等事件
  • 各种各样的delegate：其实也是基于通知的一种实现
  • GCD(Grand Central Dispatch)：异步执行任务的技术之一，抽象执行任务。 可以调度在串行队列中顺序执行的代码，或者在具有不同优先级的不同队列上同时运行多个任务
  • 闭包（Closures）：创建可以在类之间传递的分离的代码段

总之，你的应用程序的代码根据各种外部因素（如用户输入，网络活动或其他操作系统事件）而根据不同的顺序执行。 每次用户启动你的应用程序时，代码可能会以完全不同的顺序运行，具体取决于外部因素。这是无法进行预测的。

这些Apple提供的API固然好，但是当程序变得庞大、复杂，各种delegate满天飞。由于所有异步API都没有通用语言，因此阅读和理解代码以及关于其执行结果的推断会变得十分困难。

异步编程的关键术语

一般来说，RxSwift尝试解决以下问题：

状态（State），即共享可变状态（shared mutable state）

「状态」本身很难被定义。来看下面的一个例子：

When you start your laptop it runs just fine, but after you use it for a few days or even weeks, it might start behaving weirdly or abruptly hang and refuse to speak to you. The hardware and software remains the same, but what’s changed is the state. As soon as you restart, the same combination of hardware and software will work just fine once more.

The data in memory, the one stored on disk, all the artifacts of reacting to user input, all traces that remain after fetching data from cloud services — the sum of these is the state of your laptop.

管理你的app的所有状态，特别是那些在多个异步组件中共享的状态，是你之后需要学会解决的问题之一。

命令式编程（Imperative programming）

通俗的讲就是通过语句来改变程序状态，你通过命令行代码来告诉程序什么时候该做什么事情。

这里没有说明这些方法会做什么。 它们是否会更新view controller本身的某些属性？ 甚至，它们是否按正确的顺序被调用？ 也许有人无意中交换了这些方法调用的顺序，并将更改提交给Git。 现在应用程序的行为可能会因为调用顺序的变化而变得不一样。

副作用（Side effects）

Side effects是对当前范围之外的状态的任何改变。
例如，考虑上面示例中的最后一段代码。 connectUIControls()可能将一些类型的event handler附加到一些UI组件。 这会导致Side effects，因为它会更改view的状态：应用程序在执行connectUIControls()前后的行为会发生变化。

任何时间修改存储在磁盘上的数据或更新屏幕上label中的文本，都会导致Side effects。

Side effects are not bad in themselves. After all, causing side effects is the ultimate goal of any program!

以我的理解Side effects即为程序运行所产生的效果。

RxSwift试图通过以下几个概念，来解决上述列出的问题。

声明式代码（Declarative code）

声明式代码允许你定义一些行为，而RxSwift会在相关事件触发时运行这些行为，并为其提供一个不可变的隔离数据输入环境。

通过这种方式，你可以假设你在写异步代码的同时，类似简单的for循环一样，保证你在操作的数据的不可变性，并按照确定的顺序执行代码。

响应式系统（Reactive systems）

响应式系统是一个很抽象的术语，涵盖了iOS App的很多方面：

  • Responsive（响应式）：始终保持UI最新，展示最新的应用状态。
  • Resilient（适应性）：每个行为都是隔离定义的，并提供灵活的错误恢复机制。
  • Elastic（弹性）：用代码处理不同的工作负载，通常实现诸如延迟拉动数据收集，事件调节和资源共享等功能。
  • Message driven（消息驱动）：组件使用基于消息的通信来提高可重用性和隔离性。

RxSwift基础

再次给出一个定义：

RxSwift finds the sweet spot between traditionally imperative Cocoa code and purist functional code. It allows you to react to events by using immutable code definitions to process asynchronously pieces of input in a deterministic, composable way.

构成RxSwift的三个部分分别是：observables（可观察对象）、operators（运算符）、schedulers（调度器）。下面展开讲这三个部分。

Observables（可观察对象）

Observable<T>类提供了Rx代码的基础，它允许一个类在一段时间内接收另一个类发出的值。

Observable与Observer为一对多的关系，可以把Observer理解为「观察者」。Observable<T>类允许一个或多个Observer实时对任何事件做出反应，并更新应用程序UI，或以其他方式处理和使用新传入的数据。

Observable<T>类遵循的ObservableType协议本身很简单。一个Observable可以发射（和一个Observer可以接收）三种类型的事件：

  • next事件：带有最新的数据的事件。这也是Observers接收数据的方式。
  • completed事件：此事件会完成并终止事件序列。 这意味着Observable已经成功完成了其生命周期，也不会再继续发出任何事件。
  • error事件：Observable出现错误并终止，而且停止发出其他事件。

下面有两种不同的observable序列：

有限observable序列

这是一种最终会以completed事件或error事件结束的序列。

在iOS应用中，考虑从Internet下载文件的代码：

  • 首先，你开始下载并开始读取下载的数据。
  • 然后，你重复地接收大量数据，这些都是完整文件的一部分。
  • 在网络连接断开的情况下，下载将停止，连接将超时并出现错误。
  • 或者，如果成功下载所有文件的数据，它将以completed事件完成。

API.download(file:)返回一个Observable<Data>实例，它将数据值作为数据块传出。

你通过提供onNext闭包来订阅next事件。在示例中，你将数据附加到存储在磁盘上的临时文件。

你通过提供onError闭包来订阅error事件。在闭包中，你可以在控制台中显示error.localizedDescription或执行其他操作。

最后，为了处理completed事件，你提供了onCompleted的闭包，在那里你可以push一个新的视图控制器来显示下载的文件或执行其他逻辑操作。

无限observable序列

不同于文件下载或类似的活动，这些活动是最后必定终止的，还有其他的序列是无限的。 通常，UI事件是这样无限的可观察序列。

例如，考虑你需要在应用程序中对设备方向更改做出反应的代码：

  • 将你的类作为观察者订阅NotificationCenter中的UIDeviceOrientationDidChange通知。
  • 然后，你需要提供一个回调方法来处理设备方向更改。它需要从UIDevice获取当前方向，并根据最新值进行相应反应。

这种方向变化序列没有自然的结束。只要设备存在，就有可能的方向变化顺序。 此外，由于序列实际上是无限的，因此在你观察时，它始终具有初始值。

UIDevice.rx.orientation是一个虚构的控件属性，它产生一个Observable<Orientation>。你订阅它并根据当前方向更新应用程序UI。 你跳过onError和onCompleted参数，因为这些事件永远不会从该observable实例发出。
